Calendar Implementation Code Walkthrough
Honeymoon Booking System | React Frontend + Express Backend

Overview
The calendar system handles guest bookings and admin date blocking for a honeymoon property. It features:
- Monthly calendar view with availability
- Guest booking and cancellation
- Admin-only date blocking
- Conflict detection
- Role-based access control
- Real-time booking updates

  |- Calendar grid                 |- GET /api/bookings

handleDayClick(cell) - Lines 210-227
User clicks a calendar day. Three states:
- State 1: No start date yet
  Set startDate = cell.key, clear endDate
- State 2: Start set, no end yet
  If clicked date < startDate, swap them. Else set endDate.
- State 3: Both set
  Reset to State 1 (start fresh selection)

Data Models

Booking Schema (models/Booking.js)
- propertyKey (String): Unique ID for property (for example, "upcountry-honeymoon-getaway")
- propertyName (String): Display name
- guestName (String): Visitor full name
- guestEmail (String): Visitor email
- startDate (Date): Check-in (normalized to 12:00 UTC)
- endDate (Date): Check-out (normalized to 12:00 UTC)
- status (String): Enum: 'confirmed' | 'blocked' | 'cancelled'
- notes (String): Optional reservation notes
- bookedBy (Object): User snapshot: { userId, email, displayName, role }
- cancelledBy (Object): User who cancelled (if status='cancelled')
- cancelledAt (Date): When booking was cancelled
- createdAt, updatedAt (Date): Timestamps

Security and Access Control

Authentication
- All POST/DELETE routes require isAuthenticated middleware
- GET /api/bookings is public (no auth required)

Authorization
- blockDates: Admin-only (line 108)
- Cancel booking: Owner or admin (line 160)
- Book for visitor: Admin only (line 114)

Data Isolation
- All queries filter by propertyKey
- User snapshots immutable (prevent spoofing)

Error Handling
- 400: Invalid dates, missing guest info (lines 104, 113)
- 403: Unauthorized (non-admin block, not owner) (lines 108, 161)
- 404: Booking not found (line 154)
- 409: Date conflict (line 123)
- 500: Server error (lines 140, 180)

Key Design Patterns
1. Normalized Date Keys
Uses "YYYY-MM-DD" strings for reliable date comparison across timezones.

2. User Snapshots
Stores user data at booking time (immutable record of who booked).

3. Soft Delete
Bookings marked 'cancelled' not removed (preserves history).

4. Conflict Detection
MongoDB $lte/$gte on start/end dates (overlapping range query).

5. Dual-Mode Calendar
Guest view (book own) vs. Admin view (book/block for visitors).
